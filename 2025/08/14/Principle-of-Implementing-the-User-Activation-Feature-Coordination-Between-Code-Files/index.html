Principle of Implementing the User Activation Feature (Coordination Between Code Files)

发表于2025-08-14|更新于2025-08-14
|浏览量:

0. Data Structure (Laying the Foundation First)

File: database/migrations/*_create_users_table.php (or an additional migration)

  • Fields: activated (bool, default false), activation_token (string, nullable), (optional) email_verified_at (timestamp, nullable)

1. Model Responsible for “Automatically Generating Tokens”

File: app/Models/User.php

  • In the creating event, generate an activation token for the new user (unless explicitly assigned externally).

2. Routes Pass “Page Requests” to Controllers

File: routes/web.php

3. What Happens After the User Clicks “Register”?

3.1 Show Registration Page (GET /register)

File: resources/views/users/create.blade.php

  • Standard form: name, email, password → submitted to users.store.

3.2 Submit Registration (POST /users → UsersController@store)

File: app/Http/Controllers/UsersController.php

  • Validate → Save to database (activated=false, activation_token automatically included)
  • Send activation email (with link containing token)
  • Key point: Do not log in directly after successful registration; instead prompt “Go to your email to activate”.

4. What Does the Email Look Like? How Is the Link Generated?

File: app/Mail/ActivateAccount.php (Mailable)

File: resources/views/emails/activate.blade.php

This step relies on .env email configuration (MAIL_MAILER/MAIL_HOST/...). For local development, you can use Mailtrap/HELO or similar tools to receive test emails.

5. User Clicks the Link in the Email → Activation Controller Processes

File: app/Http/Controllers/ActivationController.php

6. Block Login for Non-Activated Users (Interception Point)

You’ve already written a version — the key idea is: If Auth::attempt succeeds → detect not activated → immediately log out and return a message.
File: app/Http/Controllers/SessionsController.php

You can also write an activated middleware to guard all protected pages uniformly (optional):

  • File: app/Http/Middleware/EnsureUserIsActivated.php
  • Register it in app/Http/Kernel.php and add ->middleware('activated') to the protected routes.

7. Where Does the “status” Prompt Come From?

  • In store() after registration, we used:
    return redirect()->route('login')->with('status', 'Registration successful! …');
  • In the login/reset page view:

This will display a green notification. It’s a one-time Flash message.

8. Optional Enhancements

  • Activation link expiry: Add activation_token_expires_at and validate when confirming.
  • Resend activation email: Provide a POST /email/resend endpoint to resend if not activated.
  • Use built-in email verification: Implementing MustVerifyEmail can also enforce “click the email link to use the account”; your current method uses a custom activated flag — you can choose one or combine them.

Easy-to-Understand Execution Flow (From Clicking “Register”)

  1. User fills the form → POST /users
  2. UsersController@store: Save to DB (activated=false, activation_token auto-generated) → send email → redirect to login page with status prompt
  3. User opens email and clicks the link → GET /email/confirm/{token}
  4. ActivationController@confirm: Find user → set activated=true, clear token, log in, redirect
  5. Later when user logs in: SessionsController@store double-checks and blocks (deny access if not activated)

0. 数据结构（先打好地基）

文件：database/migrations/*_create_users_table.php（或新增一次迁移）

  • 字段：activated(bool, 默认 false)、activation_token(string, 可空)、（可选）email_verified_at(timestamp, 可空)

1. Model 负责“自动生成令牌”

文件：app/Models/User.php

  • 在 creating 事件里给新用户生成激活令牌（除非外部已经显式赋值）

2. 路由把“页面请求”接给控制器

文件：routes/web.php

3. 用户点击“注册”之后，代码怎么跑？

3.1 展示注册页（GET /register）

文件：resources/views/users/create.blade.php

  • 普通表单：name、email、password → 提交到 users.store

3.2 提交注册（POST /users → UsersController@store）

文件：app/Http/Controllers/UsersController.php

  • 验证 → 写库（activated=false，已自动带上 activation_token）
  • 发送激活邮件（里面有带 token 的链接）
  • 关键点：不要在注册成功后直接登录，而是提示“去邮箱激活”

4. 邮件长什么样？怎么生成链接？

文件：app/Mail/ActivateAccount.php（Mailable）

文件：resources/views/emails/activate.blade.php

这一步依赖 .env 邮件配置（MAIL_MAILER/MAIL_HOST/...）。本地开发可用 Mailtrap/HELO 等收信工具。

5. 用户点邮件中的链接 → 激活控制器处理

文件：app/Http/Controllers/ActivationController.php

6. 未激活时禁止登录（拦截点）

你已经写了一个版本，关键思路是：Auth::attempt 成功 → 发现未激活 → 立刻登出并返回提示。
文件：app/Http/Controllers/SessionsController.php

也可以写一个 中间件 activated 给需要登录的页面统一把关（可选）：

  • 文件：app/Http/Middleware/EnsureUserIsActivated.php
  • 在 app/Http/Kernel.php 注册并给受保护路由加上 ->middleware('activated')

7. “status” 提示从哪里来？

  • 上面的 store() 注册成功后用了：return redirect()->route('login')->with('status', '注册成功！…');

  • 登录页/重置页视图里这段：

    就会显示绿色提示。它是 一次性的 Flash 消息。

8. 可选增强

  • 激活链接有效期：加 activation_token_expires_at，确认时顺带校验。
  • 重发激活邮件：提供一个 POST /email/resend，如果未激活就重发。
  • 使用框架自带邮箱验证：实现 MustVerifyEmail 也能达到“点邮件链接才能用”的效果；你当前方案是“自定义 activated 开关”，两者可以二选一或组合。

一眼看懂的执行顺序（从点击“注册”开始）

  1. 用户填表单 → POST /users
  2. UsersController@store：写库（activated=false、activation_token 自动生成）→ 发送邮件 → 重定向到登录页 + status 提示
  3. 用户打开邮箱点链接 → GET /email/confirm/{token}
  4. ActivationController@confirm：查人 → 置 activated=true、清空 token、可登上并跳转
  5. 日后用户登录时：SessionsController@store 再次兜底拦截（未激活就不让过）
